STAGE 03 · WEEK 07 — 09 平台化

LLMOps 应用平台可视化

构建 LLMOps 可视化管理平台，实现 Prompt 模板管理、流式响应展示与用户授权认证体系，让 AI 应用可管理、可监控。

Jinja2 SSE JWT RBAC Prompt Engineering
OVERVIEW

阶段概述

当 ChatBot 具备记忆、RAG 与 Agent 能力后，下一个挑战是"可管理性"。本阶段将构建一个可视化的 LLMOps 管理平台，让 Prompt、模型响应与用户权限都成为可控的资产。

第七周聚焦 Prompt 模块——设计模板数据模型、变量注入与版本管理，支持 A/B 测试与可视化编辑；第八周深入响应模块——SSE 流式协议、Redis 响应缓存、多模型切换与降级、监控仪表盘。

第九周搭建授权认证体系——用户注册登录、JWT 双 Token 机制、RBAC 权限模型与安全加固。最终交付一个具备 Prompt 管理 + 流式响应 + 认证的 LLMOps 可视化管理平台。

FOR YOU

与你经验的对照

可迁移的技能
  • 平台化管理 → 你做过报表平台、数据中台，Prompt 管理平台概念上完全一致
  • 调度编排 → DataWorks 调度经验直接迁移到 Prompt 模板版本管理和 A/B 测试
  • 用户认证 → 你设计过 rds 配置表和后端接口，JWT 认证是标准 Web 安全方案
需要补课的内容
  • SSE 流式响应深入 → 不只是概念，需要处理断线重连、心跳保活等工程细节
  • JWT + RBAC → 你可能没做过完整的认证体系，需要学习双 Token 机制
  • 前端权限守卫 → Vue Router 的路由级权限控制，需要前端知识
学习建议
  • 这个阶段是你的舒适区——把 LLMOps 平台当成"带 LLM 引擎的 DataWorks"来理解
  • Prompt 模板管理 = SQL 模板管理，版本控制逻辑一样
  • 重点放在 SSE 和 JWT 上，这两个是需要补课的工程技能

WEEKLY BREAKDOWN

每周详细内容

W7
WEEK 07

Prompt 模块

Prompt 模板管理

设计 Prompt 模板数据模型（模板名 / 内容 / 变量 / 版本号），实现模板 CRUD 接口。支持模板分类与标签管理。

PromptTemplate: id · name · content · variables[] · version category · tags[] · created_at · updated_at

变量注入与渲染

实现 Jinja2 模板引擎变量注入，支持系统变量（日期 / 用户名）与自定义变量。设计变量校验机制。

from jinja2 import Template tpl = Template("你好 {{user}}，今天是 {{date}}") rendered = tpl.render(user="张三", date="周一")

A/B 测试与效果对比

实现 Prompt 模板版本管理，支持多版本对比测试。设计效果评估指标（响应质量评分 / 用户满意度 / Token 消耗），用数据驱动 Prompt 迭代。

Prompt 可视化编辑器

开发在线 Prompt 编辑器，支持变量高亮、实时预览、历史版本回滚。让运营人员无需改代码即可调整 Prompt。

WEEK 08

响应模块

SSE 流式响应

深入理解 Server-Sent Events 协议，实现 Flask SSE 端点。处理连接断开重连、心跳保活机制。

@app.route("/api/v1/chat/stream") def stream(): def generate(): for chunk in llm.stream(prompt): yield f"data: {chunk}\n\n" return Response(generate(), mimetype="text/event-stream")

响应缓存策略

基于 Redis 实现响应缓存，相同 Prompt + 参数直接返回缓存结果。设计缓存失效策略（TTL / 主动失效），降低重复调用的 Token 成本。

多模型切换

支持 OpenAI GPT-4 / GPT-3.5 / Claude 等多模型接入，实现模型动态切换与降级策略（主模型超时自动切换备选模型），保障服务可用性。

响应监控面板

开发响应耗时监控、Token 使用统计、错误率仪表盘，让平台运营状态可视化、可预警。

授权认证

用户注册登录

实现用户注册 / 登录 / 找回密码流程，密码 bcrypt 加密存储。支持邮箱验证与第三方登录（GitHub / Google OAuth）。

JWT Token 认证

实现 JWT Token 签发与校验，Access Token + Refresh Token 双 Token 机制。处理 Token 过期与自动续签。

Access Token → 前端内存 · 短时效 (15min) Refresh Token → HttpOnly Cookie · 长时效 (7d) # Access 过期时用 Refresh 换新

RBAC 权限管理

设计角色-权限模型（管理员 / 开发者 / 普通用户），实现 API 级别权限控制。前端路由级权限守卫，做到接口与界面双重防护。

安全加固

接口限流（Rate Limiting）、SQL 注入防护、XSS 过滤、CSRF 防护，构建多层纵深防御体系。

TECHNICAL HIGHLIGHTS

技术要点

Jinja2 模板引擎

使用 Jinja2 实现 Prompt 变量注入与渲染，支持系统变量与自定义变量校验。

SSE 协议

基于 text/event-stream 实现服务端推送，配合心跳保活与断点续传。

JWT 双 Token 认证

Access Token 短时效存内存，Refresh Token 长时效存 HttpOnly Cookie，自动续签。

RBAC 权限模型

角色-权限映射实现 API 级与路由级双重控制，支撑多租户场景。

FAQ

常见问题

Q1

SSE 连接经常断开？

实现 Last-Event-ID 机制支持断点续传，添加心跳包（每 15s 发送注释行），Nginx 配置 proxy_buffering off 避免缓冲堆积。

Q2

JWT Token 存哪里？

Access Token 存内存（前端 Vuex / Pinia），Refresh Token 存 HttpOnly Cookie，避免 XSS 攻击窃取。

Q3

Prompt 版本管理怎么做？

每次修改创建新版本，保留历史版本，支持 diff 对比与一键回滚，配合 A/B 测试用数据驱动迭代。
