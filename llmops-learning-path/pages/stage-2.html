OVERVIEW

阶段概述

基础 ChatBot 只能依赖单轮上下文，无法记住用户偏好，也无法检索私有知识。本阶段将赋予它三个核心能力：长期记忆、知识检索与工具调用，使其从"能对话"升级为"能用"。

第四周聚焦记忆模块，通过 Redis 实现短期与长期记忆，结合滑动窗口与摘要机制突破上下文长度限制；第五周搭建 RAG 数据管道，完成文档解析、分块、Embedding 向量化与 Faiss 相似度检索，实现基于知识库的精准回答。

第六周引入 Agent 与 ReACT 推理模式，让模型能够自主决策何时检索知识、何时调用外部工具，最终交付具备记忆 + RAG + Agent 三位一体能力的商业级 ChatBot。

FOR YOU

与你经验的对照

可迁移的技能
  • Redis 缓存 → 你在联通项目用过 Redis 缓存用户点击页面，记忆模块的 Redis 用法类似
  • 文档处理管道 → RAG 的"文档解析→分块→向量化→检索"本质就是 ETL，你做了5年
  • Faiss 向量检索 → 概念上类似 ClickHouse 的分布式检索，只是用向量相似度替代 SQL 条件
需要补课的内容
  • Embedding 模型 → 理解文本如何变成向量，选型策略（BGE/M3E/OpenAI）
  • ReACT Agent 框架 → 从确定性数据流转到 LLM 自主决策，这是思维方式的转变
  • Function Calling → OpenAI 的工具调用接口规范，需要理解 JSON Schema 定义
学习建议
  • Week 4 记忆模块对你很简单，Redis 你已经会用，重点理解对话上下文管理策略
  • Week 5 RAG 是核心——用你做 ETL 的思维理解：文档分块 = 数据拆分，向量化 = 数据转换，检索 = 数据查询
  • Week 6 Agent 是本阶段难点，建议多写几遍 ReACT 循环代码加深理解

WEEKLY BREAKDOWN

每周详细内容

W4
WEEK 04

记忆模块

对话上下文管理

理解对话上下文窗口限制，实现滑动窗口策略保留最近 N 轮对话。设计对话摘要机制——当对话超过窗口时自动摘要前文，确保关键信息不丢失。

滑动窗口保留最近若干轮原始对话，超出部分由 LLM 生成结构化摘要注入系统提示，兼顾上下文连贯性与 Token 成本。

Redis 长期记忆

搭建 Redis 服务，设计记忆存储结构（短期记忆：最近对话列表；长期记忆：用户偏好摘要）。实现记忆的读写与过期策略。

# Redis 记忆结构 short_term:{user_id} → List[message] # 最近 N 轮 long_term:{user_id} → String(summary) # 偏好摘要 TTL: short_term = 1h · long_term = 7d

多轮对话优化

实现对话历史压缩、关键信息提取、上下文相关性排序。处理多用户并发会话隔离，确保不同用户的记忆互不污染。

W5
WEEK 05

RAG 数据模块

文档处理管道

支持 PDF / Word / Markdown 文档解析，实现文档分块策略（固定长度分块、语义分块、递归分块）。理解 chunk_size 和 overlap 参数对检索效果的影响。

Embedding 向量化

选择 Embedding 模型（OpenAI text-embedding-3-small / BGE / M3E），理解向量维度与质量的关系。批量向量化文档块。

from openai import OpenAI client = OpenAI() resp = client.embeddings.create( model="text-embedding-3-small", input=chunks # List[str] ) vectors = [d.embedding for d in resp.data]

向量数据库 Faiss

搭建 Faiss 本地向量索引，理解 IVF、HNSW 索引类型。实现相似度检索（余弦相似度 / L2 距离），返回 Top-K 相关文档块。

RAG 生成流程

检索相关文档 → 构造增强 Prompt（context + question）→ 调用 LLM 生成回答。实现引用标注，标注回答来源的文档段落。

def rag_answer(question, top_k=3): docs = faiss.search(embed(question), k=top_k) context = "\n".join(d.text for d in docs) prompt = f"参考资料：{context}\n问题：{question}" return llm.chat(prompt), docs # 返回引用

WEEK 06

工具功能 (Agent)

ReACT 推理模式

理解 ReACT（Reasoning + Acting）框架——Thought → Action → Observation 循环。设计 Agent 的推理链与工具选择逻辑。

Thought: 用户问今天天气，我需要调用搜索工具 Action: search_weather(city="北京") Observation: 晴, 26℃ Thought: 已获得结果，可以回答用户 Final Answer: 北京今天晴，气温 26℃。

自定义工具定义

定义搜索工具、计算器工具、数据库查询工具等。使用 OpenAI Function Calling 或 LangChain Tool 接口规范工具定义，明确参数 schema 与返回格式。

Agent 多步编排

实现 Agent 多步推理与工具调用循环，处理工具调用失败与异常。设置最大迭代次数防止死循环，并在超出时优雅降级。

Agent 与 RAG 结合

让 Agent 能自主决定何时检索知识库、何时调用外部工具，实现智能路由。这是从"被动问答"到"主动解决问题"的关键跃迁。

TECHNICAL HIGHLIGHTS

技术要点

Redis 缓存策略

短期记忆用 List、长期记忆用 String，配合 TTL 与摘要压缩，平衡召回率与成本。

Faiss 向量索引

使用 IVF 或 HNSW 索引加速近邻搜索，支持余弦相似度与 L2 距离度量。

Embedding 模型选型

中文场景优先 BGE-large-zh / M3E，英文场景可用 OpenAI text-embedding-3，需权衡维度与存储成本。

ReACT Agent 编排

Thought-Action-Observation 循环驱动工具调用，结合 max_iterations 防止死循环。

FAQ

常见问题

Q1

RAG 检索效果不好怎么办？

优化分块策略（减小 chunk_size）、尝试语义分块、增加 overlap、使用 reranker 重排序检索结果，并校验 Embedding 模型与语种匹配。

Q2

Agent 工具调用陷入死循环？

设置 max_iterations（建议 5-10 次），添加终止条件检测，实现 fallback 策略——超过阈值后直接返回当前最佳结果。

Q3

Embedding 模型怎么选？

中文场景推荐 BGE-large-zh 或 M3E，英文场景推荐 OpenAI text-embedding-3，注意向量维度与存储成本的权衡。
